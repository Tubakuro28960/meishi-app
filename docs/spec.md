# 名刺アプリ 詳細設計書 v1

## 1. 文書概要

### 1-1. システム名
名刺OCR・メール送信支援アプリ

### 1-2. 目的
ユーザーが撮影またはアップロードした名刺画像から、氏名・会社名・メールアドレスなどの情報を抽出し、
その情報を一覧として保存・管理し、テンプレートを用いてメールを作成できるようにする。

さらに、以下の2つの送信方法に対応する。

- **即時送信モード**：ユーザーがその場でメールアプリを開き、内容確認後に送信する
- **予約送信モード**：ユーザーが指定した日時になると、サーバー側で自動送信する

本設計書は、過去の実装で起きた問題を踏まえ、最初から破綻しにくい構成で設計することを目的とする。

---

## 2. システムのゴール

本システムで実現したいことは以下の通り。

1. 1枚または複数枚の名刺画像をアップロードできる
2. 1枚の写真内に複数の名刺が写っていても対応できる
3. OCRで名刺情報を抽出できる
4. 抽出結果をユーザーが確認・修正できる
5. 読み取った名刺情報をリストとして保存できる
6. メールテンプレートを作成・編集・保存できる
7. 即時送信の場合はメールアプリを開いて、すぐ送れる
8. 予約送信の場合は指定日時に自動で送信される
9. 送信履歴や送信予定を確認できる
10. 失敗時に再送や状態確認ができる

---

## 3. 前提方針

### 3-1. 基本方針
本システムは以下の考え方で設計する。

- OCR処理、保存、テンプレート管理、予約送信を責務ごとに分離する
- 即時送信と予約送信は別の処理フローとして扱う
- ユーザーが後から修正・確認できるよう、OCR結果は必ず保存する
- 自動送信はクライアントではなくサーバー側で行う
- 失敗時のリカバリができるよう、送信ジョブの状態を管理する

### 3-2. 重要な設計上の結論

#### 即時送信
即時送信は、**ユーザー端末のメールアプリを開く方式**とする。
この場合、最終的な送信操作はユーザーが行う。

#### 予約送信
予約送信は、**ユーザーが指定した日時に、サーバーが自動送信する方式**とする。
この機能を成立させるには、単なる mailto リンクでは不可能であり、
サーバー側に以下が必要になる。

- 送信予約データの保存
- 定期実行処理（cron / scheduled job）
- メール送信機能（Resend / SendGrid / Gmail API など）
- 送信状態管理

このため、予約送信機能を本当に実現するには、**バックエンド必須**である。

## 5. 機能要件

## 5-1. 認証機能

### 概要
ユーザーごとに名刺情報、テンプレート、送信履歴を分離するため、ログイン機能を持たせる。

### 要件
- メールアドレス＋パスワードログイン
- 将来的にGoogleログイン追加可能な設計
- ユーザーごとにデータを分離
- 他人の名刺データやテンプレートを閲覧できない

### 備考
認証を入れないと、複数ユーザー利用時にデータ混在のリスクが高い。
そのため、初期版でも認証は必須機能とする。

---

## 5-2. 名刺画像アップロード機能

### 概要
ユーザーは名刺画像をアップロードする。

### 入力形式
- 単一画像
- 複数画像
- 1枚の写真に複数の名刺が含まれるケースにも対応

### 要件
- jpg / jpeg / png を許可
- 複数ファイル同時アップロード可能
- ドラッグ＆ドロップ対応
- スマホ撮影画像にも対応
- アップロード後にプレビュー表示

### バリデーション
- 非対応拡張子は拒否
- サイズ上限を設ける
- 画像未選択時は送信不可

---

## 5-3. 名刺検出・切り出し機能

### 概要
1枚の画像の中に複数枚の名刺が写っている場合、それぞれを個別画像として切り出す。

### 要件
- 名刺領域を自動検出
- 検出した各名刺を個別に切り出し
- 必要に応じて回転補正
- 検出数をユーザーに表示
- 誤検出時は再調整できる余地を残す

### 想定課題
- 背景と名刺の境界が曖昧な場合
- 名刺が重なっている場合
- 斜め撮影時の歪み

### 方針
初期版では「自動検出＋簡易補正」を採用し、精度問題が大きければ将来的に手動トリミング機能を追加する。

---

## 5-4. OCR機能

### 概要
切り出した名刺画像から文字を抽出し、項目ごとに構造化する。

### 抽出対象
- 氏名
- 会社名
- 部署名
- 役職
- メールアドレス
- 電話番号
- 住所
- Webサイト
- 備考

### 要件
- OCR結果をそのまま全文保持する
- 構造化した項目を別で保持する
- メールアドレスは優先的に高精度抽出
- 氏名・会社名など曖昧な項目は後で修正可能にする

### 方針
OCR結果は100%正しい前提にしない。
必ず「OCR生データ」と「ユーザー修正後データ」の両方を保持する。

---

## 5-5. OCR結果確認・編集機能

### 概要
OCRで読み取った名刺情報を、ユーザーが画面上で修正できる。

### 要件
- 項目ごとに編集可能
- 名刺画像を見ながら修正可能
- 必須項目（特にメールアドレス）が空なら警告表示
- 修正後に保存可能

### 必須に近い理由
OCRは必ず誤読が発生するため、この編集画面は必須とする。

---

## 5-6. 名刺リスト保存機能

### 概要
読み取った名刺を一覧形式で保存・管理する。

### 一覧で表示する項目
- 氏名
- 会社名
- メールアドレス
- 登録日
- 最終更新日
- タグ
- 送信状態（未送信 / 送信済み / 予約あり など）

### 要件
- 一覧表示
- 検索
- ソート
- 詳細表示
- 編集
- 削除
- タグ付け

### 検索例
- 名前で検索
- 会社名で検索
- メールアドレスで検索
- タグで絞り込み

---

## 5-7. テンプレート管理機能

### 概要
メール本文や件名のテンプレートを作成・編集・削除できる。

### 管理項目
- テンプレート名
- 件名テンプレート
- 本文テンプレート
- 使用可能な差し込み項目
- 作成日
- 更新日

### 差し込み例
- {{name}}
- {{company}}
- {{department}}
- {{position}}
- {{email}}
- {{sender_name}}

### 要件
- テンプレート新規作成
- テンプレート編集
- テンプレート削除
- 複製
- デフォルトテンプレート設定
- プレビュー表示

### 備考
テンプレート編集機能は今回の必須要件。
本文だけでなく件名も編集対象に含める。

---

## 5-8. 即時送信機能

### 概要
ユーザーがその場で送る場合の機能。

### 要件
- テンプレート適用後の件名・本文を生成
- 宛先ごとに送信用内容を作成
- mailto または端末連携でメール作成画面を開く
- ユーザーが最終確認して送信できる

### 制約
- 実際の送信ボタンはユーザー操作になる
- 大量件数だとタブ数が多くなる可能性あり

### 補足
この方式は簡易で安全だが、完全自動送信ではない。

---

## 5-9. 予約送信機能

### 概要
ユーザーが指定した日時に、自動的にメールを送信する機能。

### 要件
- 送信日時の指定
- タイムゾーン考慮
- 予約ジョブの作成
- 指定時刻に自動送信
- 成功/失敗状態の記録
- 送信前のキャンセル
- 送信前の編集
- 再送

### 必要な内部機能
- scheduled_jobs テーブル
- cron またはスケジューラ
- メール送信API
- 送信ログ
- リトライ制御

### 重要事項
予約送信を実現するには、送信元メールアドレスの扱いを決める必要がある。

想定方式は以下のどちらか。

#### 方式A：サービス共通の送信元アドレスから送る
- 実装しやすい
- 自動送信しやすい
- ただしユーザー本人のアドレスから送ったようには見えない

#### 方式B：ユーザー自身のメールアカウントと連携して送る
- Gmail API / Outlook API などが必要
- 実装難易度が高い
- ユーザー本人名義で送れる

### 初期版推奨
まずは **方式A（サービス側送信）** を前提に設計し、
将来的に Gmail API 連携を拡張できる構造にする。

## 5-10. 送信履歴・送信予定管理機能

- 予定日時

- 予約済みジョブのキャンセル

## 6. 非機能要件

## 6-1. セキュリティ
- ユーザーごとのデータ分離
- 認証必須
- API保護
- 機密情報の環境変数管理
- 送信対象メールアドレスのログ管理
- 不正送信防止

## 6-2. 可用性
- 送信予約ジョブが落ちても再実行できる
- 一時的な送信失敗時はリトライ可能

## 6-3. 保守性
- OCR処理、テンプレート処理、送信処理を分離
- テーブル責務を明確にする
- 将来的に送信プロバイダ変更可能にする

## 6-4. 拡張性
- Googleログイン追加
- Gmail API連携追加
- 手動トリミング追加
- CSV出力追加
- 一括送信の高度化

## 9. 処理フロー設計

## 9-1. OCR保存フロー
1. 画像アップロード
2. ストレージ保存
3. 名刺検出
4. 切り出し画像生成
5. OCR実行
6. 構造化抽出
7. OCR結果確認画面表示
8. ユーザー修正
9. business_cards に保存

---

## 9-2. 即時送信フロー
1. 名刺選択
2. テンプレート選択
3. 差し込み後の件名・本文生成
4. send_jobs に immediate で記録
5. メールアプリ起動
6. ユーザーが送信
7. 必要に応じて「手動で送信済みに更新」または送信起動時点で履歴化

### 注意
mailto だけでは「本当に送信されたか」はシステム側で厳密に保証しにくい。
そのため、状態管理は「起動済み」または「ユーザー確認ベース」になる可能性がある。

---

## 9-3. 予約送信フロー
1. 名刺選択
2. テンプレート選択
3. 差し込み済み件名・本文生成
4. scheduled_at を設定
5. send_jobs に pending で保存
6. 定期実行バッチが pending ジョブを取得
7. 実行対象を processing に更新
8. メール送信APIで送信
9. 成功なら sent / 失敗なら failed に更新
10. send_logs に記録

---

## 10. 送信方式設計

## 10-1. 即時送信
### 実装候補
- mailto リンク
- Gmail compose URL
- Outlook compose URL

### 推奨
初期版では mailto ベースまたは compose URL ベース。

### メリット
- 実装が簡単
- ユーザーが最終確認できる
- 誤送信リスクを下げやすい

### デメリット
- 完全自動ではない
- 一括件数が多いと扱いづらい

---

## 10-2. 予約送信
### 実装候補
- Resend
- SendGrid
- Gmail API

### 初期版推奨
- まずは Resend などのトランザクションメール基盤
- 将来的に Gmail API 連携を検討

### 理由
Gmail API は柔軟だが、OAuth、トークン更新、送信権限管理が必要で難易度が上がる。
まずは安定実装を優先する。

6. **ユーザーごとのデータ分離のため認証を必須にする**

## 18. まとめ

今回のアプリは、単なるOCRアプリではなく、

- 名刺情報の抽出
- データ保存
- テンプレート管理
- 即時送信
- 予約自動送信

までを一気通貫で扱う業務支援アプリである。

そのため、最初から

- データ構造
- 送信方式
- 認証
- ジョブ管理
- エラー処理

を分けて考えることが非常に重要である。

特に予約送信は「メールアプリを開くだけ」では実現できず、
バックエンド・スケジューラ・送信APIを組み合わせた仕組みが必要になる。

本設計書では、その前提を踏まえて、実装可能で破綻しにくい構成として整理した。
